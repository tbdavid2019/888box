# 開發 TODO

這份文件整理目前後續開發最重要的兩個方向，之後改版時可直接依此拆任務。

## 1. 繁體中文化 / 多語系規劃

### 目標

- 先完成整站繁體中文化。
- 再評估是否升級為可切換語系架構。

### 現況

- 目前文案大多直接寫死在 PHP、HTML、JS 中。
- 目前頁面語系標記多為 `zh-CN`。
- 目前沒有語言包、翻譯字典、語系切換器或 i18n helper。

### 第一階段：先做固定繁體中文版本

- 盤點所有使用者可見文案位置。
- 將前台、後台、安裝頁、提示訊息全部改為繁體中文。
- 將 `lang="zh-CN"` 改為適合的繁中標記，例如 `zh-Hant` 或 `zh-TW`。
- 統一用詞，例如：圖片、連結、儲存、設定、帳號、密碼、上傳成功、上傳失敗。
- 檢查 README、安裝流程、系統提示、錯誤訊息是否同步繁中化。

### 第二階段：如果要做真正多語系

- 設計統一的翻譯資料結構，例如 `lang/zh-Hant.php`、`lang/zh-CN.php`、`lang/en.php`。
- 建立 PHP 端文字存取 helper，例如 `t('upload.success')`。
- 建立前端 JS 文案字典或由後端注入前端語系資料。
- 將模板與 JS 中硬編碼文案逐步替換成語系 key。
- 加入語系選擇策略，例如：設定檔、Cookie、Session、網址參數。
- 確保 API 回傳訊息是否也需要依語系切換。

### 建議優先順序

- 先做固定繁中化。
- 等功能穩定後，再做多語系抽離。

## 2. API 圖床化 / 外部應用整合規劃

### 目標

- 讓其他應用可以穩定把這套系統當作圖床服務使用。
- 明確定義 API 使用方式、授權方式與回應格式。

### 現況

- 目前已有 `api.php` 可處理圖片上傳。
- 已支援 Token 驗證與來源網域限制。
- 成功上傳後已會回傳圖片 URL，可供外部應用直接使用。
- 目前比較像可用的上傳端點，不是完整文件化的公開 API 平台。

### 必做項目

- 整理正式 API 文件。
- 明確定義請求方式、表單欄位、授權 header、錯誤格式。
- 固定成功回應欄位，建議外部應用優先使用 `data.url`。
- 明確定義失敗回應結構與錯誤碼。
- 補上 cURL、JavaScript、PHP 的串接範例。

### 建議強化項目

- 補上 API 版本概念，例如 `/api/v1/upload` 對應到 `api.php`。
- 明確定義 CORS 與 `site_domain` 的設定方式。
- 評估是否增加 API key 管理機制，而不只依賴單一 Token。
- 評估是否增加上傳頻率限制、IP 限制與安全日誌。
- 評估是否需要提供刪除、查詢、健康檢查等額外 API。
- 評估回傳格式是否加入 MIME、storage、created_at 等欄位。

### 外部整合方向

- Markdown 編輯器 / 筆記工具
- 部落格或 CMS
- 留言系統
- 後台管理系統
- 自動化工作流服務

### 建議優先順序

- 先把上傳 API 文件化與穩定化。
- 再補授權、安全與更多管理型 API。

## 3. 安全性修復（GitHub Issue #30）

來源：<https://github.com/JLinMr/PixPro/issues/30>

### 目標

- 重新驗證 issue #30 提出的高風險漏洞。
- 修正目前程式碼中仍然存在的安全問題。
- 將安全修復納入後續改版優先工作。

### 目前初步核對結果

- `config/delete.php` 的未授權刪除問題，看起來目前版本已補上登入檢查。
- `api.php` 的 `validateToken()` 仍然存在白名單網域可跳過 Token 驗證的風險，需要優先處理。
- `config/storage.php` 的 S3 設定仍有 `verify => false`，需要修正。
- `config/database.php` 仍有 `mkdir(..., 0777, true)`，需要修正權限。
- `admin/login.php` 的密碼重設流程仍需重新審視安全性。
- `config/upload.php` 仍使用 `mt_rand()` 產生檔名，可評估改為更安全的隨機來源。

### 必做項目

- 逐條重驗 issue #30 的 6 個漏洞是否在目前版本仍然成立。
- 修正 `api.php` 的授權繞過問題，避免僅憑 Referer 白名單即可上傳。
- 修正 `config/storage.php` 的 SSL 驗證設定。
- 修正所有過寬目錄權限設定。
- 重新設計密碼重設機制，避免只憑使用者名稱就能重設。
- 檢查敏感憑證的保存方式，評估改走環境變數或加密存放。

### 建議追加項目

- 補上 CSRF 防護。
- 加強檔案型別與內容驗證。
- 補強安全日誌與異常行為紀錄。
- 為安全修復建立專門的驗證清單。
